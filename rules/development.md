# 開発プロジェクトの進め方

## 開発フロー（ウォーターフォール）

本プロジェクトは **ウォーターフォール** で進める。上流から順に、各フェーズの成果物を確定・承認してから次へ進む。
**ユーザーはお客様兼サポーター** — 各フェーズ完了時にユーザー（お客様）の承認を得てから次フェーズに着手する。

| # | フェーズ | 成果物 | 完了条件（承認の目安） |
|---|---------|--------|----------------------|
| 1 | 案件書 | `document/proposal.md` | 何を作るか・目的・ターゲット・予算・目標売上・制約・KPI が埋まり、ユーザー承認済み |
| 2 | 要件定義 | `document/requirements.md` | 案件書の内容が機能要件／非機能要件に落ち、ユーザー承認済み |
| 3 | 基本設計 | `document/basic-design.md` + `document/diagrams/`（architecture・ER図） | 画面・機能・外部IF・データ概念が確定 |
| 4 | 詳細設計 | `document/detailed-design.md` + `document/diagrams/`（クラス図・シーケンス図） | クラス・メソッド・処理ロジック・DB物理が確定 |
| 5 | 開発 | `src/` | 詳細設計どおりに実装、コミット済み |
| 6 | テスト | `tests/` | テストが通り、ユーザーが動作確認できる状態 |

- **案件書はユーザーへの対話ヒアリングで作成する**（`/dev-setup` 直後に実施）
- 前フェーズの成果物が確定するまで次フェーズの成果物を書き始めない
- 後工程で前提が崩れたら、該当する上流フェーズに戻って成果物を修正する（戻りも記録する）

下記「開発手順」「PoC 判断基準」「計画→レビューサイクル」は、各フェーズ内で使う技法として併用する。

## 開発手順（必須順序）

1. **要件定義** — `document/requirements.md` を確定
2. **機能検証（PoC）** — 要件をもとに重要機能の実現可能性を実際に確認
   - PoC コードは `poc/{機能名}/` に throwaway として配置（本実装には流用しない）
   - 結果サマリは `document/feature/{機能名}.md` の「機能検証ログ」欄に転記
   - NG → 要件定義に戻る、OK → 仕様作成へ進む
3. **基本設計** — `document/basic-design.md` `document/feature/*.md` を作成（検証済み内容のみ）
4. **詳細設計・図表作成** — `document/detailed-design.md` と `document/diagrams/` のクラス図・ER図・構成図・シーケンス図
5. **実装** — `src/` `tests/`
   - 仕様書通りに実装する
   - 仕様書に書かれていない判断・変更・妥協・意思決定が発生したら、**全て** `document/implementation-notes.md` に逐次追記する（後でまとめてはダメ）
6. **テスト**

2〜5は 計画 → レビュー（最大N回） → 実装 → チェック → コードレビュー（最大N回） → プロジェクト最適化 のサイクルを回す。

**仕様 = 確定事項。検証で不可なら仕様にせず要件に戻すこと。**
**要件定義書、仕様書、図表はHTMLファイルで出力してユーザーが確認しやすいようにする**

## PoC 必須／スキップ判断基準

**PoC 必須:**
- 新規ライブラリ / SDK 採用
- 外部 API 依存（OAuth・Webhook・サードパーティ連携）
- 性能要件あり（レスポンス < Nms / N TPS 等）
- 暗号化・認証・決済など細部が複雑
- 過去経験ない技術領域

**PoC スキップ可:**
- 単純な CRUD
- 標準的な画面遷移のみ
- 既存実装の流用

## PoC 検証パターン

| 検証対象 | 方法 |
|---------|------|
| 技術可能性 | 最小スクリプトで動作確認 |
| 外部連携 | スタブ叩いて疎通確認 |
| 性能 | ベンチスクリプトで実測 |
| データモデル | sqlite 等で SQL 試打 |
| アルゴリズム | テストデータで検算 |
| UI/UX | HTML モック or Figma でスクショ |

## アーキテクチャ・設計の選び方

設計書を作成する前に、以下の基準をもとにアーキテクチャ・設計パターンを選択する。

### 選択基準

| 観点 | 内容 |
|---|---|
| **規模** | 小規模ならシンプルな構成（MVC・レイヤード）、大規模なら分離を強める |
| **ドメイン複雑度** | 業務ロジックが複雑 → DDD + クリーン／ヘキサゴナルアーキテクチャ |
| **読み書き比率** | 読み取りが高負荷 → CQRS（必要に応じてイベントソーシングも検討） |
| **チーム規模** | 大チーム・複数チーム → マイクロサービス／境界づけられたコンテキスト |
| **フロント種別** | Web・モバイルなど複数クライアント → BFF パターンを検討 |
| **変更頻度** | 仕様変更が多い → 依存方向を内側に統一し、外部を差し替えやすくする |
| **テスト容易性** | テストしやすさを重視 → ヘキサゴナル（Port & Adapter）が有効 |

### 規模別の目安

```
小規模・シンプル    → MVC / レイヤード + SOLID原則
業務ロジックが複雑  → DDD + クリーンアーキテクチャ or ヘキサゴナル
読み取りが高負荷    → CQRS（+ イベントソーシング）
サービス間連携      → マイクロサービス + BFF
```

### 原則（規模問わず共通）

- **SOLID** — 単一責任・開放閉鎖・リスコフ置換・インタフェース分離・依存逆転
- **DRY** — 重複を排除する
- **KISS** — シンプルに保つ
- **YAGNI** — 今不要なものは作らない
- **GoFデザインパターン** — 生成・構造・振る舞いの23種から必要に応じて採用

過度な設計（Over-engineering）を避ける — 現時点の要件に必要な最小限の複雑さにとどめる。アーキテクチャの選択理由を設計書に明記する。

## 実装ルール（仕様書通り + implementation-notes）

- **仕様書（`document/requirements.md` `document/basic-design.md` `document/detailed-design.md` `document/feature/*.md` `document/tech-stack.md`）通りに実装する**
- 仕様書 = 確定事項。勝手に拡張・省略・解釈変更しない
- 仕様書に書かれていないことが出てきた瞬間、即 `document/implementation-notes.md` に記録する

### implementation-notes に記録するもの

| 種別 | 例 |
|------|-----|
| 仕様外の判断 | 仕様未記載の挙動を実装上どう決めたか |
| 仕様との差分 | 仕様と異なる実装にした理由 |
| 妥協点 | 性能・互換性・ライブラリ制約で諦めた事項 |
| 採用したライブラリ・パターン | 仕様未記載の技術選定 |
| 却下した代替案 | 検討したが採らなかったもの、その理由 |
| 一時的 workaround | 後で直す前提のハック |

判断した**その場**でログ追記（コミットと同タイミング推奨）。後でまとめて書く運用は禁止。

### コードコメント — なぜその実装にしたか

AI にコードを書かせるうえで最も大切なのは「**なぜその実装にしたか（WHAT ではなく WHY）**」を残すこと。
コードを読めば「何をしているか」は分かるが、「なぜそうしたか」はコメントがないと失われる。

- **自明でない設計判断にはコメントで理由を残す** — なぜこのアルゴリズム・データ構造・ライブラリ・パターンを選んだか
- **WHAT（何をしているか）ではなく WHY（なぜそうしたか）を書く** — コードが語る情報は繰り返さない
- 特にコメントすべき箇所:

  | 対象 | コメントに書く内容 |
  |------|------|
  | 非自明なアルゴリズム選択 | なぜ単純解でなくこれを選んだか（性能・制約など） |
  | 直感に反する実装 | 一見おかしく見えるが必要な理由 |
  | workaround / ハック | 何を回避しているか、正攻法を採れない理由、解消条件 |
  | マジックナンバー・定数 | その値の根拠・出典 |
  | 意図的に「やらない」選択 | あえて実装しなかった理由（YAGNI 等） |
  | 仕様・外部制約への対応 | どの要件・API 仕様・互換性に由来するか |

- コメントは**コードと同じコミットで**書く。後付けしない
- 重大な設計判断は `document/implementation-notes.md` にも記録する（コードコメント = 局所的な WHY、implementation-notes = プロジェクト全体の意思決定ログ）

`document/implementation-notes.md` の「ログ」セクションに日付つきで追記：

```markdown
### YYYY-MM-DD — {タイトル / 対象機能}

- **コンテキスト:** 何をしていて、どこで判断が必要になったか
- **選んだもの:** 実際に採用した実装・設定・ライブラリ
- **理由:** なぜそれを選んだか
- **却下した代替案:** 検討したが選ばなかったもの、その理由
- **影響範囲:** どのファイル・どの機能に効くか
- **見直しトリガー:** どういう条件になったら再検討するか（任意）
- **関連:** 関連 Issue / PR / コミット（任意）
```

PR レビュー時は **仕様書 + implementation-notes の差分** を必ず確認する。仕様書になく implementation-notes にもない実装上の判断があれば、追記してもらってからマージする。

## 開発中のルール

- **テスト駆動開発(TDD)をする** — 先にテストを書いてから実装する
- **推測でコードを書かない**
- **アプリは必ずGitHubのプライベートリポジトリで管理する**
- **ファイルの編集・作成・削除を行ったら必ずコミットする**
- **タスクを更新する**
- **変更を行ったことをユーザーが実際に確認できる状態にする** — PRを出した後に起動してユーザーが確認できる状態にすること
- **仕様書・要件定義書・設計書・クラス図などのドキュメントはHTMLファイルで出力する** — `/doc-html` スキルを使うこと（`/doc-html <path>` で特定ファイル、引数なしで `document/` 配下を一括変換）
- **PlanModeでプランを確定する前に `plan-view` スキルを実行すること** — `ExitPlanMode` を呼び出す直前に `plan-view` スキルを使い、`document/PLAN.html` としてHTML出力してブラウザで表示する

## ブランチ戦略

- `main`（本番）/ `develop`（開発）/ `feature/*`（機能）/ `fix/*`（バグ修正）/ `release/*`（リリース）
- **mainブランチは基本操作しない** — developブランチからのマージのみ
- **developブランチから必ずfeature/○○・fix/○○ブランチを切って作業する**
- **1機能 = 1ブランチ** — 常に動作する状態を維持する
- **機能マージ時はPRをdevelopに出してレビューする** — マージ後はタスクを更新する
- `feature/*` `fix/*` : PR マージ後に自動削除（GitHub `--delete-branch-on-merge` 設定済み）
- ローカルの残骸は `git fetch --prune` で定期同期

### タスク開始ルール

1. `develop` から作業ブランチを切る
   - 機能追加: `feature/issue-{Issue番号}-{タスク名（kebab-case）}`
   - バグ修正: `fix/issue-{Issue番号}-{タスク名（kebab-case）}`
2. `tasks.md` の該当タスクを `[ ]` → `[着手中 YYYY-MM-DD]` に変更
3. 実装 → ファイル操作のたびにコミット
4. タスク完了 → `tasks.md` の該当タスクを `[x]` に変更
5. **Stop hook が自動で** feature/fix ブランチを push + develop への PR を作成する
6. **PR のマージはユーザーが行う**

## GitHub Issues 連携

- **各タスクはGitHub Issuesで管理する** — Issueに紐づいたブランチを切り、PRでIssueをクローズする
- `tasks.md` にはIssueへのリンクを記載する

## 図表同期更新ルール

`document/` 配下を更新したら `document/diagrams/` の関連図表も同一コミットで更新する：

- 新機能追加 → クラス図・シーケンス図 更新
- データモデル変更 → ER図 更新
- アーキテクチャ変更 → architecture.md 更新

図表は Mermaid 形式（GitHub・Obsidian で直接レンダリング可）。

## README 更新ルール

`document/` 配下を更新して以下に変化があった場合は README.md も更新すること：
- 主要機能の追加・削除・名称変更
- 技術スタックの変更
- 動作環境・セットアップ手順の変更
- プロジェクト概要・目的の変更

## プロジェクト最適化（サイクル末尾で実施）

各イテレーション末尾に実施する：

- **ルール更新** — `CLAUDE.md` のルール・制約を実態に合わせて修正
- **フック最適化** — `.claude/settings.json` のフック（PreToolUse / PostToolUse / Stop 等）をプロジェクトの技術スタック・テスト・lint コマンドに合わせて調整（`/verify-setup` 活用）
- **権限チューニング** — 頻繁に承認を求められる read-only 操作を allowlist に追加（`/fewer-permission-prompts` 活用）
- **タスク整理** — `tasks.md` / GitHub Issues を現状に同期し、完了済み・不要タスクをクローズ
