# 開発プロジェクトの進め方

## 開発手順（概要）

1. 要件定義書の作成（ユーザーからのヒアリングをもとに）
2. 設計書の作成（要件定義書をもとに、機能設計・画面設計・DB設計など）
3. コーディング（設計書をもとに実装）
4. テスト（単体テスト・結合テスト・総合テストなど）
5. デプロイ（本番環境へのリリース）

2-4は下記ステップで繰り返す。
計画 → レビュー（最大N回） → 実装 → チェック → コードレビュー（最大N回） → ルール更新というサイクルを回す。

## アーキテクチャ・設計の選び方

設計書を作成する前に、以下の基準をもとにアーキテクチャ・設計パターンを選択する。

### 選択基準

| 観点 | 内容 |
|---|---|
| **規模** | 小規模ならシンプルな構成（MVC・レイヤード）、大規模なら分離を強める |
| **ドメイン複雑度** | 業務ロジックが複雑 → DDD + クリーン／ヘキサゴナルアーキテクチャ |
| **読み書き比率** | 読み取りが高負荷 → CQRS（必要に応じてイベントソーシングも検討） |
| **チーム規模** | 大チーム・複数チーム → マイクロサービス／境界づけられたコンテキスト |
| **フロント種別** | Web・モバイルなど複数クライアント → BFF パターンを検討 |
| **変更頻度** | 仕様変更が多い → 依存方向を内側に統一し、外部を差し替えやすくする |
| **テスト容易性** | テストしやすさを重視 → ヘキサゴナル（Port & Adapter）が有効 |

### 規模別の目安

```
小規模・シンプル    → MVC / レイヤード + SOLID原則
業務ロジックが複雑  → DDD + クリーンアーキテクチャ or ヘキサゴナル
読み取りが高負荷    → CQRS（+ イベントソーシング）
サービス間連携      → マイクロサービス + BFF
```

### 原則（規模問わず共通）

- **SOLID** — 単一責任・開放閉鎖・リスコフ置換・インタフェース分離・依存逆転
- **DRY** — 重複を排除する
- **KISS** — シンプルに保つ
- **YAGNI** — 今不要なものは作らない
- **GoFデザインパターン** — 生成・構造・振る舞いの23種から必要に応じて採用

### 注意

- 過度な設計（Over-engineering）を避ける — 現時点の要件に必要な最小限の複雑さにとどめる
- アーキテクチャの選択理由を設計書に明記する

## 開発中のルール

- **テスト駆動開発(TDD)をする** — 先にテストを書いてから実装する
- **推測でコードを書かない**
- **アプリは必ずGitHubのプライベートリポジトリで管理する** — 環境構築後にプライベートリポジトリを作成してプッシュする
- **ファイルの編集・作成・削除を行ったら必ずコミットする**
- **タスクを更新する**
- **変更を行ったことをユーザーが実際に確認できる状態にする** — PRを出した後に起動してユーザーが確認できる状態にすること

### ブランチ戦略

- `main`（本番）/ `develop`（開発）/ `feature/*`（機能）/ `fix/*`（バグ修正）/ `release/*`（リリース）
- **mainブランチは基本操作しない** — developブランチからのマージのみ
- **developブランチから必ずfeature/○○・fix/○○ブランチを切って作業する**
- **外部公開はmainから切ったrelease/ブランチで行う**
- **1機能 = 1ブランチ** — 常に動作する状態を維持する
- **機能マージ時はPRをdevelopに出してレビューする** — マージ後はタスクを更新する

### GitHub Issues 連携

- **各タスクはGitHub Issuesで管理する** — Issueに紐づいたブランチを切り、PRでIssueをクローズする
- `tasks.md` にはIssueへのリンクを記載する
