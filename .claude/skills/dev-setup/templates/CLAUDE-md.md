## 開発フロー（必須順序）

1. **要件定義** — `document/requirements.md` を確定
2. **機能検証（PoC）** — 要件をもとに重要機能の実現可能性を実際に確認
   - PoC コードは `poc/{機能名}/` に throwaway として配置（本実装には流用しない）
   - 結果サマリは `document/feature/{機能名}.md` の「機能検証ログ」欄に転記
   - NG → 要件定義に戻る、OK → 仕様作成へ進む
3. **仕様作成** — `document/spec.md` `document/feature/*.md` を作成（検証済み内容のみ）
4. **設計・図表作成** — `document/diagrams/` のクラス図・ER図・構成図・シーケンス図
5. **実装** — `src/` `tests/`
   - **仕様書通りに実装すること**
   - 仕様書に書かれていない判断・変更・妥協・意思決定が発生したら、**全て** `document/implementation-notes.md` に逐次追記すること（後でまとめてはダメ）
6. **テスト**

**仕様 = 確定事項。検証で不可なら仕様にせず要件に戻すこと。**

### PoC 必須／スキップ判断基準

**PoC 必須:**
- 新規ライブラリ / SDK 採用
- 外部 API 依存（OAuth・Webhook・サードパーティ連携）
- 性能要件あり（レスポンス < Nms / N TPS 等）
- 暗号化・認証・決済など細部が複雑
- 過去経験ない技術領域

**PoC スキップ可:**
- 単純な CRUD
- 標準的な画面遷移のみ
- 既存実装の流用

### PoC 検証パターン

| 検証対象 | 方法 |
|---------|------|
| 技術可能性 | 最小スクリプトで動作確認 |
| 外部連携 | スタブ叩いて疎通確認 |
| 性能 | ベンチスクリプトで実測 |
| データモデル | sqlite 等で SQL 試打 |
| アルゴリズム | テストデータで検算 |
| UI/UX | HTML モック or Figma でスクショ |

## 実装ルール（仕様書通り + implementation-notes）

実装フェーズ（src/ tests/ への着手以降）は以下を厳守する：

### 原則

- **仕様書（`document/requirements.md` `document/spec.md` `document/feature/*.md` `document/tech-stack.md`）通りに実装する**
- 仕様書 = 確定事項。勝手に拡張・省略・解釈変更しない
- 仕様書に書かれていないことが出てきた瞬間、即 `document/implementation-notes.md` に記録する

### implementation-notes に記録するもの（全て）

| 種別 | 例 |
|------|-----|
| 仕様外の判断 | 仕様未記載の挙動を実装上どう決めたか |
| 仕様との差分 | 仕様と異なる実装にした理由 |
| 妥協点 | 性能・互換性・ライブラリ制約で諦めた事項 |
| 採用したライブラリ・パターン | 仕様未記載の技術選定 |
| 却下した代替案 | 検討したが採らなかったもの、その理由 |
| 一時的 workaround | 後で直す前提のハック |

### 記録タイミング

- 判断した**その場**でログ追記（コミットと同タイミング推奨）
- 後でまとめて書く運用は禁止（忘れる・抜ける）
- 仕様書側を直すべきと判断したら、仕様書を更新したうえで「{日付}: {仕様書名} に反映」と implementation-notes に残す

### 記録フォーマット

`document/implementation-notes.md` の「ログ」セクションに、日付つきで以下を1エントリ追記：

```markdown
### YYYY-MM-DD — {タイトル / 対象機能}

- **コンテキスト:** 何をしていて、どこで判断が必要になったか
- **選んだもの:** 実際に採用した実装・設定・ライブラリ
- **理由:** なぜそれを選んだか
- **却下した代替案:** 検討したが選ばなかったもの、その理由
- **影響範囲:** どのファイル・どの機能に効くか
- **見直しトリガー:** どういう条件になったら再検討するか（任意）
- **関連:** 関連 Issue / PR / コミット（任意）
```

### レビュー時

- PR レビュー時は **仕様書 + implementation-notes の差分** を必ず確認する
- 仕様書になく implementation-notes にもない実装上の判断があれば、追記してもらってからマージする

## 図表同期更新ルール

`document/` 配下を更新したら `document/diagrams/` の関連図表も同一コミットで更新する：

- 新機能追加 → クラス図・シーケンス図 更新
- データモデル変更 → ER図 更新
- アーキテクチャ変更 → architecture.md 更新

図表は Mermaid 形式（GitHub・Obsidian で直接レンダリング可）。

## ブランチ削除ルール

- `feature/*` `fix/*` : PR マージ後に自動削除（GitHub `--delete-branch-on-merge` 設定済み）
- `develop` `main` `release/*` : 削除しない
- ローカルの残骸は `git fetch --prune` で定期同期

## タスク開始ルール

タスクを開始するときは必ず以下の手順で進めること：

1. `develop` ブランチから作業ブランチを切る
   - 機能追加: `feature/issue-{Issue番号}-{タスク名（kebab-case）}`
   - バグ修正: `fix/issue-{Issue番号}-{タスク名（kebab-case）}`
   - 例: `feature/issue-3-user-authentication`
2. tasks.md の該当タスクを `[ ]` → `[着手中 YYYY-MM-DD]` に変更
3. 実装 → ファイル操作のたびにコミット
4. タスク完了 → tasks.md の該当タスクを `[x]` に変更
5. **Stop hook が自動で** feature/fix ブランチを push + develop への PR を作成する
6. **PR のマージはユーザーが行う**

## README 更新ルール

`document/` 配下を更新して以下に変化があった場合は README.md も更新すること：
- 主要機能の追加・削除・名称変更
- 技術スタックの変更
- 動作環境・セットアップ手順の変更
- プロジェクト概要・目的の変更
